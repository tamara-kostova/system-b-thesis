## 1. Introduction

The EHDS Regulation (EU) 2025/327 Chapter IV mandates that secondary use of electronic health data across EU member states occurs through a permit-based system with secure processing environments (SPEs), output disclosure checking, and audit logging. No open-source reference implementation of this workflow exists.

**System B** implements the full Chapter IV lifecycle as five composable microservices, validated against the OMOP CDM v5.4 standard using 1,119 synthetic patients generated by Synthea. The thesis contribution is a reproducible reference implementation with an LLM-assisted discovery interface.

## 2. Architecture

The system is structured as five phases, each a separate service:

```
Researcher
    │
    ├──► Phase 1: Discovery API (:8003) ─────────────────────┐
    │        Public, read-only. Concept search, counts.      │
    │        All counts suppressed at < 10 (EHDS Art. 50).   │
    │                                                         ▼
    ├──► Phase 5: LLM Gateway (:8006)          Phase 5 calls Phase 1 tools only.
    │        Natural-language interface to Phase 1.
    │
    ├──► Phase 2: Permit Service (:8002)
    │        Application → review → grant/refuse (EHDS Arts. 67–68).
    │        State machine: draft→submitted→under_review→granted|refused→expired.
    │
    │    [On grant]
    │        │
    │        ▼
    ├──► Phase 3: SPE Provisioner (:8004)
    │        Creates permit-scoped Postgres views (anonymized/pseudonymized).
    │        Launches JupyterLab in an isolated Docker network.
    │
    │    [Researcher analyses data in SPE]
    │        │
    │        ▼
    └──► Phase 4: Output Airlock (:8005)
             Automated disclosure checks → human reviewer → approved download.
```

### Key design decisions

**Database-level access control.** On permit grant, Phase 3 creates a Postgres schema `permit_<id>` with views that filter to the permitted concept IDs (via `CONCEPT_ANCESTOR` descendants), time window, and patient subset. A dedicated Postgres user with `search_path` set to this schema physically cannot query outside it — enforcement is in the database, not application code.

**Single-function small-cell suppression.** `apps/discovery_api/suppression.py` contains one function returning `"<10"` for counts below 10. All count-returning endpoints import this function. It is never duplicated.

**Docker network isolation.** Each SPE container connects to an `--internal` Docker network with no default route. The Postgres container has an alias on this network; internet egress from the SPE requires iptables rules not present in this prototype (documented limitation).

## 3. Implementation

### Phase 0 — Synthetic Data Pipeline

Synthea generates synthetic patient data; `sql/synthea_to_omop.py` converts it to OMOP CDM v5.4. Dataset: 1,119 persons, 39,279 conditions, 48,451 drug exposures, 309,418 measurements. SNOMED codes exceeding Postgres INTEGER range are clamped to 0 (a known Synthea quirk). The vocabulary is self-referential only — no ATHENA download required, keeping the system fully reproducible.

### Phase 1 — Discovery API

FastAPI service (port 8003). Four endpoints: dataset catalogue, concept search (ILIKE over `cdm.concept`), concept descendants (via `CONCEPT_ANCESTOR`), and patient counts. Every count passes through `suppress()` before returning.

### Phase 2 — Permit Service

FastAPI backend with two Streamlit UIs (applicant and reviewer). The `Permit` Pydantic model captures purpose (one of six EHDS Article 53 categories), data scope (domains + concept IDs + time window), and format (anonymized or pseudonymized). `PermitStateMachine` enforces legal transitions and calls `log_event()` on every change. Illegal transitions raise `IllegalTransitionError`.

### Phase 3 — SPE Provisioner

For each granted permit, the provisioner creates:
1. A Postgres schema with domain-filtered views (conditions, drugs, measurements).
2. An internal Docker network connecting the SPE container to Postgres.
3. A JupyterLab container with credentials scoped to the permit schema.

Notebook saves are audit-logged via a JupyterLab post-save hook. A starter notebook (`spe_starter.ipynb`) is pre-loaded with the DB connection snippet and an egress verification cell.

### Phase 4 — Output Airlock

Automated disclosure checks run on every submission:

| Check | Catches |
|-------|---------|
| `small_cell` (CSV) | Any integer count 1–9 |
| `id_column` (CSV) | All-unique columns with identifier-like names |
| `image_ocr` (PNG/JPG) | > 100 unique numbers in OCR text |
| `json_schema` (JSON) | Arrays of objects or nested row data |

Files failing any check are `blocked` immediately. Files passing enter `pending_review` for human sign-off before download. Every event is audit-logged.

### Phase 5 — LLM Gateway

*In progress.* Two modes: Mode A (pre-permit, calls Phase 1 tools only) and Mode B (in-SPE copilot). Guardrails: concept IDs only from tool returns, no row-level tool, PII-in-prompt rejection. Backends: Anthropic Claude, OpenAI GPT-4, Ollama/Llama 3.2 (local, offline).

## 5. Evaluation

### Test suite

| File | Tests | Validates |
|------|-------|-----------|
| `test_suppression.py` | 5 | Small-cell suppression cannot be bypassed |
| `test_state_machine.py` | 11 | Only legal permit transitions allowed |
| `test_airlock_checks.py` | 19 | Disclosure checks catch documented attacks |
| `test_projection.py` | 12 | Permit-scoped SQL is generated correctly |

All 47 tests pass.

### Performance

Measured on a local development machine (Intel i7, Docker Desktop):

| Operation | Time | Notes |
|-----------|------|-------|
| SPE provisioning (`POST /spe`) | 327 ms | Docker network create + container start + Postgres user/view setup |
| Airlock submission + automated checks (`POST /submissions`) | 75 ms | CSV small-cell + ID-column check on a 100-row file |

Both operations are fast enough for interactive use. SPE provisioning is a one-time cost per permit; the 327 ms includes creating the Docker network, connecting Postgres, launching the JupyterLab container, and setting up the permit-scoped schema. The 75 ms airlock figure covers multipart upload parsing, running all four check functions, and persisting the result to Postgres.

### Adversarial exfiltration attempts

| Attack | Blocked by |
|--------|-----------|
| CSV with count = 3 | `small_cell` check |
| CSV with `pseudo_id` column, all unique | `id_column` check |
| JSON array of patient records | `json_schema` check |
| Row data nested inside aggregate JSON | `json_schema` check |
| Binary `.pkl` model file | `unknown_format` check |
| Screenshot with > 100 numbers | `image_ocr` check (requires Tesseract) |

## 6. Limitations and Future Work

| Limitation | Production path |
|------------|----------------|
| Flat OMOP vocabulary (no ATHENA) | Integrate ATHENA vocabulary for real concept hierarchy traversal |
| SPE internet access not fully blocked | Add iptables `OUTPUT DROP` rules at container start |
| Audit log not tamper-evident | Append-only Postgres table or WORM storage |
| No researcher authentication | Integrate eIDAS / institutional SSO |
| Single-node deployment | Kubernetes with persistent volumes |
| Ollama/llama3.1:8b has inconsistent tool-call compliance | Use Anthropic Claude or a larger model for reliable guardrail enforcement |
